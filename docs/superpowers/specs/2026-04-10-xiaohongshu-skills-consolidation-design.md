# Xiaohongshu Skills Consolidation Design

**Date:** 2026-04-10  
**Repo:** `/Users/fulln/opensource/xiaohongshu-skills`  
**Goal:** 将当前本机安装版 `~/.claude/skills/xiaohongshu-skills` 中已经实际生效的小红书 skills、发布链路、copy-ready 约束、长文链路、账号能力与文案规范，完整整合回源码仓库，并将该源码仓库切换为用户个人 public 仓库的唯一维护源。

---

## 1. 背景与问题

当前存在两套事实来源：

1. **本机安装版**：`~/.claude/skills/xiaohongshu-skills`
   - 是 Claude Code 当前真实生效的 skill 实现。
   - 已经包含多账号、copy-ready 发布、长文发布、发布前确认、系列资产真源等更完整的规则。

2. **源码仓库版**：`/Users/fulln/opensource/xiaohongshu-skills`
   - 是希望长期维护和公开发布的仓库。
   - 但它与安装版已经出现文档、边界和能力描述漂移。

这导致了几个问题：

- 安装版与仓库版不一致，后续修改容易重复劳动。
- 仓库 README / 根 SKILL / 子 SKILL 对外说明不完全覆盖用户当前真实工作流。
- `xhs-publish` 与 `xhs-content-ops` 的职责边界不够清楚。
- `copy-ready`、系列发布真源、台账式发布控制等现实工作流没有完整沉入源码仓库。
- 仓库当前远端仍是他人组织仓库，不适合作为用户个人长期公开维护源。

---

## 2. 本次整合目标

本次不是重做技能体系，而是做一次**以源码仓库为唯一真源的完整收束**。

### 2.1 必达目标

1. 把安装版中当前真实生效的 skills 定义与关键规则收回到源码仓库。
2. 统一根 `SKILL.md`、5 个子 `SKILL.md`、`README.md`、`CLAUDE.md`、必要的架构/说明文档。
3. 正式纳入当前真实使用的发布工作流：
   - `copy-ready` 单文件发布真源
   - 长文模式链路
   - 发布前用户确认
   - 取消发布时保存草稿
   - 系列内容与发布台账约束
4. 增补“可信、具体、亲切、不像 AI 模板文”的风格规则，落在合适的 skill 层。
5. 将仓库远端切换为用户个人 public 仓库，作为后续唯一公开维护源。

### 2.2 明确不做

1. 不大规模重构 CLI 架构。
2. 不把风格规则硬编码到 parser 或命令参数校验里。
3. 不拆出新的 skill 子目录（本轮仍保持 5 个子 skill）。
4. 不保留继续推送到 `autoclaw-cc/xiaohongshu-skills` 的关系。

---

## 3. 设计原则

### 3.1 源码仓库成为唯一真源

整合完成后，`/Users/fulln/opensource/xiaohongshu-skills` 应成为唯一维护源：

- 文档更新先改源码仓库。
- 之后如需更新本机安装版，应从源码仓库同步，而不是双向散改。
- 后续新增功能、规则和工作流，一律以源码仓库为基准。

### 3.2 保持当前 5 个子 skill，不额外拆分

保留现有结构：

- `xhs-auth`
- `xhs-publish`
- `xhs-explore`
- `xhs-interact`
- `xhs-content-ops`

原因：

- 用户已经在当前结构上完成了真实使用与验证。
- 当前问题主要是“漂移”和“职责模糊”，不是 skill 数量不对。
- 本轮应优先提升一致性，而不是引入新认知成本。

### 3.3 文档与真实链路对齐优先于形式美化

本轮整合优先让外部文档准确表达真实行为，而不是追求概念上更漂亮的结构。凡是已经在安装版和真实工作流中成立的链路，应明确沉入源码仓库。

## 4. 目标结构与职责边界

### 4.1 根 `SKILL.md`

**职责：**
- 作为统一入口和一级路由器。
- 负责全局约束、总览、路由规则。
- 不承担具体执行细节。

**需要补齐的内容：**
- 与安装版一致的路由语义。
- 对 `copy-ready` 作为发布真源的全局约束。
- 对 Bridge 模式 / 本地真实浏览器执行的说明。
- 对图片边界、确认边界、路径边界的一致说明。

### 4.2 `xhs-auth`

**职责：**
- 登录状态检查。
- 二维码登录 / 手机验证码登录。
- 多账号管理与切换。

**整合目标：**
- 保持安装版更完整的多账号工作流。
- 保留“每次登录都必须重新向用户确认手机号”的约束。
- 统一登录方式说明、二维码展示规范、失败处理。

### 4.3 `xhs-explore`

**职责：**
- 搜索发现层。
- 首页 feed、关键词搜索、详情拉取、用户主页查看。

**整合目标：**
- 对 `feed_id` + `xsec_token` 作为后续详情 / 互动输入做更明确说明。
- 统一结果呈现风格，强调结构化输出。

### 4.4 `xhs-interact`

**职责：**
- 互动执行层。
- 评论、回复、点赞、收藏。

**整合目标：**
- 明确互动必须依赖已有 `feed_id` / `xsec_token`。
- 评论 / 回复必须先经用户确认。
- 与 `xhs-explore` 的前后关系表述更清楚。

### 4.5 `xhs-publish`

**职责：**
- 最终发布执行层。
- 承担图文、视频、长文、草稿保存、copy-ready 发布、发布前确认、发布前复核。

**整合目标：**
- 正式纳入 `parse-copy-ready` / `fill-publish-copy-ready` / `publish-copy-ready`。
- 明确 `post-xx-copy-ready.md` 才是系列发布真源，`final/*.md` 不能直接当发布输入。
- 保留长文模式、模板选择、下一步填写、草稿保存等实际工作流。
- 新增“发布前风格闸门”：
  - 标题是否只有概念没有判断
  - 开头是否过泛、像行业报告
  - 正文是否缺少具体对象/动作/场景
  - 结尾是否模板化
  - 首评是否自然承接而非硬互动

### 4.6 `xhs-content-ops`

**职责：**
- 内容编排与运营层。
- 竞品分析、热点追踪、内容创作、多步互动工作流。

**整合目标：**
- 正式承接“怎么写”的规则，而不是只描述流程。
- 新增“可信、具体、亲切、不像 AI 模板文”的内容创作规范：
  - 先判断，后展开
  - 抽象观点必须落到真对象 / 真动作 / 真场景
  - 可有第一人称状态变化，但不写成日记流
  - 避免官腔、课程腔、统一模板收尾
  - 不照搬别人的黑话宇宙或命名体系
- 与 `xhs-publish` 形成清晰边界：
  - `xhs-content-ops` 负责“怎么写”
  - `xhs-publish` 负责“怎么发、发前怎么复核”

## 5. `copy-ready` 与系列发布真源设计

### 5.1 `copy-ready` 的地位

整合后要明确：

- 对已整理成系列发布资产的内容，**唯一发布真源**是 `post-xx-copy-ready.md`。
- `final/xx-最终发帖版.md` 是内容沉淀层，不是最终执行输入。
- 发布时不再去其他文档临时拼凑标题、标签、首评。

### 5.2 `copy-ready` 的字段与 parser

当前安装版 parser 已经要求：

- `发帖标题`
- `正文（可直接复制）`
- `封面文案`
- `标签`
- `首评（建议发布后立刻发）`

本轮不破坏现有 parser 结构。

### 5.3 可选扩展

可在模板说明中增加**可选章节**，如：
- `风格自检`
- `配图执行`
- `发帖前最后确认`

这些附加章节不应破坏现有 parser；只作为 skill 生成与复核的辅助信息层。

## 6. README / CLAUDE / 架构文档的收束方向

### 6.1 README

README 应面向公开用户，承担这些职责：

- 清楚说明 5 个子 skill 的能力边界。
- 准确说明安装方式。
- 说明 Bridge 架构与本地浏览器执行方式。
- 说明图文发布只接受本地图片绝对路径。
- 说明系列资产推荐走 `copy-ready` 发布链路。
- 给出最小可用示例：登录、搜索、详情、发布、互动。

### 6.2 CLAUDE.md

CLAUDE.md 应面向仓库维护者，承担这些职责：

- 指导未来如何修改 skill 而不再造成安装版 / 仓库版漂移。
- 说明何处是对外边界，何处是内部实现。
- 说明 `publish_pipeline.py` 等内部编排器不是主要对外能力入口。
- 说明 copy-ready / 风格规则 / 系列发布链路在文档中的主位置。

### 6.3 ARCHITECTURE.md

若已有架构文档，应补齐：

- Bridge 扩展模式
- CLI → bridge_server → extension → 浏览器 → 小红书 的链路
- 多账号与端口隔离
- 长文发布与普通发布的分流
- copy-ready 发布在整体架构中的位置

## 7. Git 仓库归属切换设计

当前仓库 `origin` 指向：
- `https://github.com/autoclaw-cc/xiaohongshu-skills.git`

本轮目标：
- 将 `origin` 直接切换为用户个人 public 仓库。

### 7.1 切换方式

- 不保留旧 `origin`。
- 直接将当前仓库 `origin` 改为用户个人仓库 URL。
- 后续 push 全部走用户个人仓库。

### 7.2 风险说明

- 一旦切换，当前本地仓库默认上游就不再是原项目。
- 若用户之后还想回看原项目，只能手动再加远端。

这是用户已明确接受的行为边界，本轮按此执行。

## 9. 方案选择结论

本次采用的方案是：

**方案 2：以源码仓库为唯一真源做一次完整收束。**

这是因为当前最核心的问题不是缺功能，而是：

- 安装版和源码版漂移
- 文档与真实行为不一致
- 职责边界模糊
- 真实发布工作流没有完整沉回仓库

该方案能在不大改架构的前提下，把现有真实工作流沉淀为可维护、可公开发布的用户个人仓库。
